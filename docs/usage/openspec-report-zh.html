OpenSpec 在 DimOS 里的使用汇报

OpenSpec 不是为了多写文档，而是把需求、设计、验收和实现任务变成可追踪的工程对象，尤其适合 AI 参与开发的仓库。

为什么现在需要它

DimOS 是机器人系统仓库，改动常常跨模块、蓝图、agent skills、CI 和真实硬件路径。OpenSpec 把“先想清楚再动代码”固定成流程，能减少需求漂移、隐藏假设和 AI 直接开写带来的返工风险。

一、为什么需要 OpenSpec

传统开发里，需求经常散落在聊天、issue、commit 和 PR 描述里。AI 加入后，如果没有明确规格，agent 很容易按局部代码猜意图，导致实现看似完成但边界错了。

它解决的三个核心问题

  • 对齐问题：先确定 scope、non-goals、验收标准，再写代码。
  • 上下文问题：让人和 agent 都从同一组 artifacts 开始工作。
  • 追踪问题：变更完成后可归档，后续能查到当时为什么这么设计。

二、日常使用流程

1. Explore

先把问题想清楚

适合需求还模糊、方案有分歧、需要调研代码结构时使用。

CLI: openspec list --json

2. Propose

把想法变成 change

创建 proposal、design、spec、tasks，让团队先评审范围和验收标准。

CLI: openspec new change "<name>"

3. Apply

按任务实现

实现前读取 change 的上下文文件，按 tasks.md 推进。

CLI: openspec status --change "<name>"

4. Verify

和仓库工作流对齐

DimOS 的完成标准是运行验证脚本，并把结果带到 PR 描述里。

CLI: bash scripts/verify.sh

5. Archive

完成后归档

合并或完成后归档 change，保留决策记录，避免 active changes 堆积。

三、Artifacts 各自负责什么

Artifact作用
proposal.md回答为什么做、做什么、不做什么，是对齐范围的入口。
design.md记录方案、权衡、风险和迁移思路，避免只在聊天里口头决定。
specs/沉淀能力级要求和验收标准，是后续回归、评审、归档的依据。
tasks.md把方案拆成可执行步骤，implementation 阶段逐项勾掉。

四、最佳使用模式

原则建议
适合使用 OpenSpec跨模块改动、用户可见行为、API/协议变化、机器人控制路径、CI/发布流程、需要多人评审的方案。
可以跳过 OpenSpec拼写修复、单文件小重构、注释补充、已经完全明确且风险很低的局部改动。
最佳粒度一个 change 对应一个清晰目标，最好能自然落到一个小 PR。
AI 协作方式先让 agent 读 proposal/design/spec/tasks，再实现；实现中发现假设不成立时，先更新 artifacts。
评审关注点先评审需求和边界，再评审实现；spec 是验收标准，tasks 是执行清单，design 是关键取舍记录。

建议在 DimOS 落地的标准模式

1. 变更前

对大改动先建 OpenSpec change。proposal 写清 why、scope、non-goals；design 记录关键取舍；tasks 拆到可执行。

2. 实现中

Agent 实现前读取 artifacts。实现发现需求不准或设计不成立时，先修 artifacts，再继续写代码。

3. 合并后

跑 bash scripts/verify.sh，PR 里引用 change 背景。完成后归档，保持 active changes 干净。

OpenSpec 是我们给复杂变更加上的“工程规格层”。它不替代代码、不替代 PR，而是在代码和 PR 之前先把需求、设计、验收和任务拆清楚。
