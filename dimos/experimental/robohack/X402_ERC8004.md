# x402 + ERC-8004 — what they are, and how the robot uses them

Plain-English explainer for integrating the two "agent-economy" technologies into the Patrol Dog build.
Deep technical reality (verbatim SDK shapes, addresses, faucets, pitfalls) lives in
[`research/04-agent-economy-payments.md`](research/04-agent-economy-payments.md) — read that before writing code.

**One-line framing:** **x402 is how the robot moves money; ERC-8004 is who the robot *is* and how good it
is.** Money rail + trust layer. They're independent standards that compose: x402 settles the payment,
ERC-8004 records that the job got done and how well.

> ⚠️ **Mainland-CN constraint (approach γ).** On-chain stablecoins/NFTs are a legal+cultural liability on the
> public Shanghai floor (PBoC bans extend to stablecoins/NFTs/RWA). So the *public* demo charges in **Alipay
> AI Pay**; x402 + ERC-8004 are the **"global mode"** — wired in the repo, shown **privately on testnet** to
> Dimensional's (agent-native, international) engineers. Both run on **testnet only** here; no mainnet keys on
> the venue laptop, ever.

## 1. x402 — the payment rail

### What it is
x402 revives the long-dormant HTTP status code **`402 Payment Required`** and turns it into a real payment
handshake settled in **on-chain stablecoin (USDC)**. It's a Coinbase-originated standard, now governed by the
**x402 Foundation** (a Linux Foundation effort; Cloudflare co-founded it).

### How the handshake works
1. A client `GET`s/`POST`s a resource.
2. Server replies **`402`** with a `PAYMENT-REQUIRED` header: *which network, which asset, the price, the
   pay-to address.*
3. Client signs a stablecoin transfer authorization (EIP-3009 for USDC) and **retries** with a
   `PAYMENT-SIGNATURE` header.
4. Server hands the signed payload to a **facilitator** (a hosted service that does the RPC/gas/chain
   plumbing), which verifies + settles on-chain, then returns the resource + a settlement receipt.

The facilitator is the key ergonomic win: your code never touches gas, nonces, or RPC. Testnet uses the public
`https://x402.org/facilitator` on **Base Sepolia** — **permissionless and free** (testnet USDC from faucets).
This is the whole reason it can ship in 48h where Alipay can't: **no merchant credentials.**

### Why it matters for *this* project
- **First-class Python SDK** (`pip install x402`) — same language as DimOS.
- Ships a **FastAPI middleware** *and* a **native MCP integration with a `@paid` decorator**.
- **DimOS Skills are already exposed as MCP tools** → an x402 paywall maps onto a Skill with **one decorator**.
  That's the killer seam: `@paid` on a Skill's MCP handler = the Skill now costs USDC to invoke.

### Two directions, both ~3 lines of glue
- **Robot-as-seller:** decorate `fetch()` / `patrol()` / `deliver()` with `@paid` → any agent must pay USDC to
  command the dog.
- **Robot-as-buyer:** wrap the DimOS agent's *outbound* MCP client so it auto-pays when it calls a paid
  external tool (e.g. a premium VLM/perception API), with an approval hook.

## 2. ERC-8004 — the trust layer (our home turf)

### What it is
ERC-8004 ("**Trustless Agents**", mainnet-live since Jan 2026) is **not payments.** It's the on-chain answer to
"*who is this agent, and is it any good?*" — the trust spine that makes autonomous payments *meaningful*. Three
on-chain registries:

| Registry | Purpose | Key calls |
|---|---|---|
| **Identity** (ERC-721) | The agent *is* an NFT — a metadata URI describing it + a verified payment wallet | `register(agentURI) → agentId`, `setAgentWallet(...)` |
| **Reputation** | Signed, fixed-point feedback accumulated per agent | `giveFeedback(agentId, value, decimals, tag, endpoint, …)`, `getSummary(agentId, …)` |
| **Validation** | Independent verification of work (evidence URI + response hash) | validator hooks |

Reference contracts are deployed + verified on **Ethereum Sepolia** (and Base/Linea Sepolia). Reputation is
flexible signed fixed-point: a 5-star rating is `value=5, decimals=0`; 99.77% uptime is `value=9977, decimals=2`.

> We have **prior ERC-8004 experience** — this is a *differentiator, not a stretch*. Write the
> register + feedback wiring Day-1 and make the reputation counter the visual spine.

### The minimal loop (robot earns reputation per task)
1. **Once:** robot calls `register(agentURI)` (a JSON describing the Go2 + its Skills) → gets `agentId` (an
   NFT). `setAgentWallet` binds the same wallet it transacts with on x402.
2. **Per task:** customer pays via x402 → robot runs the Skill → on success, `giveFeedback(agentId, 5, …)`.
3. **Readout:** dashboard polls `getSummary(agentId)` → a live "**Reputation: 4.8 ★ over N jobs**" counter that
   ticks up on stage. This is the single most legible on-chain artifact for a 90-second video.

## 3. How they compose

```
 Customer / judge phone  ──POST /jobs/fetch──▶  FastAPI gateway
   (x402 client; 402 → sign USDC → retry)         │  x402 middleware (Base Sepolia, x402.org/facilitator)
                                                   │  on settle → call DimOS Skill over MCP (:9990)
                                                   ▼
                                          DimOS agent → Skill: fetch()/patrol()/deliver() → Unitree Go2
                                                   │  on success (fire async, don't block the dog)
                                                   ▼
                                    ERC-8004 ReputationRegistry.giveFeedback(agentId, 5, …)  (Sepolia)
                                                   ▲
            Dashboard polls getSummary(agentId) → "Reputation: 4.8 ★ over N jobs"  ◀── hero visual
```

**x402 carries the value; ERC-8004 carries the credibility.** Payment + reputation are *both* real on-chain
artifacts a judge can verify on a block explorer — and the physical action (dog does a thing) is the emotional
payoff.

## 4. Use cases for the build

### Tier 1 — ship this (plugs straight into Direction A / Patrol Dog)
- **Pay-per-task robot (crypto twin of the vending dog).** The dog's `fetch`/`deliver`/`patrol` Skill sits
  behind an x402 paywall. A judge pays testnet USDC from their phone → dog executes. Permissionless, no creds,
  more reliable to ship in 48h than the Alipay version. *This is the recommended hero loop for "global mode."*
- **On-chain reputation counter.** Every completed task fires `giveFeedback`; the dashboard shows the dog's
  star rating ticking up live. Cheap, judge-verifiable, and visually unbeatable for the 90s video.

### Tier 2 — high wow if Tier 1 lands early
- **Self-funding robot (the inverse + closed loop).** Before fetching, the agent *spends* — pays $0.01 via
  x402 for a "premium vision" API call (which can be your *own* second paid endpoint). Now the dog **earns and
  spends** on stage: a complete economic loop, demonstrating genuine economic autonomy.
- **Trustless swarm economy (ties to Direction B).** Each Go2 has its own ERC-8004 identity + reputation;
  dogs **pay each other** for subtasks ("you scout, I'll pay you 0.05 USDC") and rate the result. Most
  ambitious; best "wow + frontier" combo — and it's exactly where our 8004 experience pays off.

### Verbal-only (mention, don't build)
- **MPP (Stripe + Tempo)** — same HTTP-402 shape, *method-agnostic* (card/wallet/crypto), back-compatible with
  x402, adds budgeted "Sessions." The answer to "*what about fiat / production?*": "MPP — same protocol, swap
  the rail." Skip building it; x402 gives the same story with zero account setup.

## 5. Make-real vs. mock (for a reliable 90 seconds)

**Make real** (cheap + judge-verifiable): the x402 `402 → pay → settle` round trip on Base Sepolia; the
ERC-8004 `register` + `giveFeedback` + `getSummary` calls on Sepolia.

**Mock / pre-stage:**
- **Don't block the dog's physical action on confirmation.** Execute the Skill the instant the facilitator
  returns `verify=ok`; let settlement finalize in the background.
- **Burner keys in env vars only.** Pre-fund testnet ETH (gas) + testnet USDC the *day before* — faucets
  rate-limit and venue Wi-Fi will fight you. Pre-warm one tx before filming.
- **Mock-facilitator fallback** behind an env flag (`FacilitatorConfig.url` → local stub that always returns
  `valid/settled`) so a network hiccup can't kill the run.
- **Fire `giveFeedback` async** after the dog finishes; show optimistic "+1 job" and reconcile on confirm.

**Graceful-failure pattern (hire signal):** if payment verification fails, the Skill should *politely refuse*
("payment not received, standing by"), not crash. Judges explicitly reward plan/execute/**fail/recover**.
