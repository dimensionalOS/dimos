# M20 Ground Upsampling Technical Plan

[中文版](/dimos/robot/deeprobotics/m20/nav/upsample_ground/README_CN.md)

## 1. Document Purpose

This document defines the design and delivery plan for a single-level ground upsampling module
for the DeepRobotics M20 navigation stack. The module fills small gaps in observed ground data
before costmap generation while preventing repeated visits from appending duplicate ground points.

The initial operating assumption is an indoor or locally planar environment, a `5 m x 5 m` local
region of interest, and a `0.05 m` output grid spacing. Multi-level maps are explicitly out of scope.

## 2. Background

The current M20 navigation pipeline is defined in
[`m20_simple_nav.py`](/dimos/robot/deeprobotics/m20/nav/m20_simple_nav.py):

1. `slam_aligned_points` provides the current point cloud in the world frame.
2. `RayTracingVoxelMap` accumulates surface voxels and publishes `global_map`.
3. `CostMapper` converts `global_map` into `global_costmap` using height gradients.
4. `ReplanningAStarPlanner` plans against the resulting costmap.

`RayTracingVoxelMap` only publishes surface points. Its internal ray misses are not represented as
free-space cells in the output `PointCloud2`. Its implementation is declared in
[`dimos/mapping/ray_tracing/module.py`](/dimos/mapping/ray_tracing/module.py), with the native update
and publication logic in
[`dimos/mapping/ray_tracing/rust/src/main.rs`](/dimos/mapping/ray_tracing/rust/src/main.rs) and
[`voxel_ray_tracer.rs`](/dimos/mapping/ray_tracing/rust/src/voxel_ray_tracer.rs).

Ground points can be sparse or removed by health gating and ray clearing. When ground points are
missing, the height-based implementation in
[`occupancy.py`](/dimos/mapping/pointclouds/occupancy.py) cannot reliably mark those cells as known
and traversable.

## 3. Goals

- Estimate the local ground surface from each current world-frame point cloud.
- Fill small gaps in well-supported ground regions at `0.05 m` spacing.
- Persist confirmed synthetic ground in world coordinates across robot motion.
- Store at most one synthetic ground point per world XY grid cell.
- Update or replace an existing ground cell when the robot revisits the same location.
- Merge synthetic ground with the latest authoritative `RayTracingVoxelMap.global_map` snapshot.
- Preserve real sensor and ray-tracing points as higher-authority data.
- Prevent synthetic ground from hiding low obstacles, walls, holes, or drop-offs.
- Keep all generated data outside the feedback path of `RayTracingVoxelMap`.
- Expose diagnostics needed to measure fitting quality, growth, latency, and rejection reasons.

## 4. Proposed Architecture

Add an `UpsampleGround` module between `RayTracingVoxelMap` and `CostMapper`.

| Port | Direction | Type | Source or consumer | Purpose |
|---|---|---|---|---|
| `current_frame` | Input | `PointCloud2` | `slam_aligned_points` | Estimate the current local ground plane |
| `global_map` | Input | `PointCloud2` | `RayTracingVoxelMap.global_map` | Authoritative accumulated real surface snapshot |
| `odometry` | Input | `Odometry` | `slam_odom` | Define the robot-relative ROI and height range |
| `global_map_upsample_ground` | Output | `PointCloud2` | `CostMapper.global_map` | Real global map fused with unique synthetic ground |
| `ground_patch` | Optional output | `PointCloud2` | Rerun or diagnostics | Visualize only the synthetic ground generated by the module |

The intended data flow is:

1. `slam_aligned_points`, `slam_odom`, and `RayTracingVoxelMap.global_map` enter separate bounded
   queues sorted by timestamp.
2. The module uses the head of the global-map queue as the reference and matches the nearest current
   frame and odometry samples.
3. A successful match triggers ground estimation and a merged output for that global map.
4. If a future sample may still arrive, the reference is retained for another attempt. If the
   reference is confirmed unsynchronizable, the current frame is skipped but the real `global_map`
   is still merged with existing historical synthetic ground and emitted as
   `global_map_upsample_ground`.
5. The current frame updates the persistent synthetic ground state.
6. The latest real global-map snapshot is fused with the synthetic ground state.
7. `CostMapper` consumes `global_map_upsample_ground` instead of the direct ray-tracing output.

The merged output must never be routed into `RayTracingVoxelMap` or back into `UpsampleGround` as
history.

## 5. Core State Model

The real `global_map` must not be accumulated by `UpsampleGround`. It is already a complete snapshot
of ray-tracing history and is used only for the current merge.

Only synthetic ground is persistent. It is stored in a world-aligned XY hash grid:

| Field | Type | Meaning |
|---|---|---|
| `ix` | `int` | `floor(world_x / ground_resolution)` |
| `iy` | `int` | `floor(world_y / ground_resolution)` |
| `z` | `float` | Current fused ground height |
| `confidence` | `float` | Confidence in the stored height |
| `observations` | `int` | Number of accepted updates, with saturation |
| `last_seen_ts` | `float` | Last accepted observation timestamp |
| `fit_id` | `int` | Identifier of the plane fit that last updated the cell |

The state key is `(ix, iy)`. Since multi-level operation is out of scope, every world XY cell has at
most one synthetic ground height.

The required invariant is:

`synthetic_ground_count == number_of_unique_world_xy_keys`

Repeated visits update an existing cell. They never append a second synthetic point at the same XY
coordinate.

## 6. Ground Estimation Algorithm

### 6.1 Input Validation and Synchronization

- Reject clouds with no finite XYZ points.
- Require `current_frame.frame_id` and `global_map.frame_id` to represent the same world frame.
- Create bounded queues for `current_frame`, `global_map`, and `odometry`, each sorted in ascending
  message-timestamp order. Insert out-of-order arrivals by timestamp rather than matching by arrival
  order.
- Always use timestamp `t_ref` from the head of the `global_map` queue as the synchronization
  reference, and run the same selection algorithm on the `current_frame` and `odometry` queues.
- Preserve `global_map.ts` and `global_map.frame_id` on the merged output.

Apply the following matching rules to each auxiliary queue:

1. If the queue is empty, return `WAIT`. Do not estimate ground or consume the reference
   `global_map`.
2. If the head timestamp equals `t_ref`, select the head sample directly.
3. If the head timestamp is less than `t_ref`, scan forward to the first sample whose timestamp is
   greater than `t_ref`.
4. If no sample greater than `t_ref` exists by the end of the queue, a future sample may not have
   arrived yet. Return `WAIT`, retain the reference `global_map`, and retry when new data arrives.
5. When the first sample greater than `t_ref` is found, compare its absolute time distance with that
   of the immediately preceding sample and select the closer one. Select the earlier sample on an
   exact tie to keep the result deterministic.
6. If the queue head is greater than `t_ref`, select it only when
   `queue_head.ts - t_ref <= max_future_sync_delta_s`, initially `0.05 s`; otherwise return
   `UNSYNCHRONIZABLE`.

Proceed to ROI filtering, plane estimation, and synthetic-state updates only after both
`current_frame` and `odometry` select a sample. If either returns `WAIT`, wait without publishing the
reference. If either returns `UNSYNCHRONIZABLE`, generate no current-frame ground and do not modify
persistent synthetic-ground state. Instead, merge the reference real `global_map` with existing
historical synthetic ground, publish `global_map_upsample_ground`, and pop the reference. This fallback
merge still applies real-point-first 3D deduplication and obstacle-conflict checks so historical
synthetic points cannot overwrite current real points. After a successful match or confirmed
synchronization failure, remove samples too old to participate in any future match and pop the
processed reference `global_map`.

The synchronization-failure output is:

`global_map_upsample_ground(t) = real_global_map(t) union validated_historical_synthetic_ground(t-1)`

It contains no synthetic point generated or updated from the current frame. If synchronized inputs are
insufficient to recompute the boundary, the most recent valid boundary may clip historical synthetic
points. Current real-point conflict checks remain mandatory even when the boundary cannot be updated.

### 6.2 Robot-Relative Region of Interest

- Center the ROI on the synchronized robot pose.
- Start with a `5 m x 5 m` XY region.
- Express the candidate height as `point.z - robot.z` rather than using an absolute world Z.
- Configure minimum and maximum relative heights from recorded M20 data.
- Remove NaN, Inf, zero-range, and obvious isolated outliers.
- Downsample candidates before fitting so near-range sensor density does not dominate the fit.

### 6.3 Robust Z Histogram Seeding

- Use a low percentile, initially 2% to 5%, rather than the absolute lowest point as the histogram
  origin.
- Bin candidate points by Z using `z_bin_size_m`, initially `0.05 m`.
- Select the dominant bin as a seed, not as the final ground classification.
- Add the immediately lower and upper bins to the fit candidate set.
- Do not accept an adjacent bin solely because its point count exceeds 50% of the dominant bin.
- Use geometric plane residual, XY connectivity, and normal consistency to decide whether adjacent
  points belong to the same surface.

This histogram step is intended to reduce the RANSAC search space. It must not determine the final
ground plane by itself.

### 6.4 Plane Fitting

- Fit a plane in the form `z = a*x + b*y + c` using RANSAC.
- Refine accepted RANSAC inliers with robust or weighted least squares.
- Calculate inlier count, inlier ratio, RMSE, and XY coverage.
- Reject line-like point sets and point sets concentrated in too few XY cells.
- Reject fits that do not include or connect to a supported region near the robot.

A fit is accepted only if every configured quality gate passes. A rejected frame produces no new
synthetic ground and leaves existing state unchanged.

### 6.5 Explored Global-Map Boundary

The explored boundary is the outer no-go boundary that upsampling must not cross; it is not a
requirement that every XY cell already contain a real point. Unknown cells inside this boundary may
still receive synthetic ground when plane support and obstacle checks pass. This fills ground that has
not yet been directly hit by the sensor while preventing unconstrained ground behind walls or other
occluding obstacles. Before every fusion, use the robot pose synchronized with `global_map` and perform
these steps:

1. Compute `relative_z = point.z - robot.z` for every `global_map` point.
2. Keep only points satisfying `relative_z_min_m <= relative_z <= boundary_relative_z_max_m`, with
   an initial fixed `boundary_relative_z_max_m` of `+0.50 m`.
3. Project retained points onto XY and rasterize them at `ground_resolution_m` to extract the outer
   boundary formed by walls, obstacles, or the end of the observed region.
4. Close only gaps no larger than `boundary_max_gap_fill_m`. Do not use an unconstrained convex hull
   that bridges into space behind walls, outside corridors, or large occluded regions.
5. Preserve valid projected connected regions independently and extract their outer boundaries;
   interior unknown holes are not no-go regions.
6. Accept a candidate when it is inside or on an allowed boundary, even if its XY cell has no real
   `global_map` point. Reject candidates that cross the outer boundary.

The final synthetic-ground generation region is the intersection of the current plane support, the
area inside the global-map outer boundary, and the obstacle-conflict-safe region. Unknown cells inside
the boundary may therefore be completed by the support mask.

Boundary validation applies to the entire persistent synthetic-ground state, not only newly generated
points. A previously stored synthetic cell crossing the current outer boundary must not be emitted. It
may remain internal pending later confirmation or be removed by expiry policy. If the outer boundary is
empty, has too few points, or cannot be extracted, emit no synthetic ground and publish only the real
`global_map` unchanged.

### 6.6 Spatial Support Mask

The module must not fill the complete `5 m x 5 m` rectangle unconditionally.

- Rasterize accepted plane inliers into the world XY ground grid.
- Select the largest connected component associated with the local ground support.
- Fill only small internal holes whose diameter is below `max_hole_fill_m`.
- Allow an outward dilation of at most `max_support_dilation_m`.
- Preserve large holes and discontinuities outside the accepted plane-support propagation. A boundary-
  interior unknown cell must not be rejected solely because `global_map` has no direct hit there.
- Stop support propagation across height discontinuities or known obstacle columns.
- Reject cells outside the observed sensor support even if the fitted infinite plane crosses them.

### 6.7 Uniform Ground Generation

- Generate one candidate point per supported world XY cell.
- Place XY coordinates on a fixed world lattice so revisits produce the same keys.
- Compute Z from the accepted plane equation at the lattice coordinate.
- Use `ground_resolution`, initially `0.05 m`, for both the state key and output spacing.
- Preserve a separate diagnostic representation of synthetic points when possible.

For a fully supported `5 m x 5 m` area, `0.05 m` spacing produces approximately 10,000 candidate
cells. The support mask should normally reduce this number.

## 7. Persistent Ground Update Rules

For each generated candidate cell:

1. Insert it when `(ix, iy)` is not present.
2. Fuse it with the existing height when the difference is no greater than
   `max_ground_update_delta_m`.
3. Increase confidence and observation count after a consistent update.
4. Replace it only when the new fit has sufficiently higher confidence and the height difference is
   larger than the normal fusion threshold.
5. Reject an inconsistent update when neither the fusion nor replacement rule passes.
6. Refresh `last_seen_ts` only after an accepted update.

Height fusion should be confidence weighted. Observation counts and confidence must saturate so an
old cell can still be corrected after later high-quality observations.

Optional confidence decay may remove old synthetic cells after `ground_stale_after_s`. A value of
zero disables time-based expiry for static environments. Explicit contradictory observations must
still be able to remove or replace stale synthetic ground.

## 8. Real and Synthetic Map Fusion

Each output is rebuilt from the latest real snapshot and the current synthetic ground state:

`output(t) = real_global_map(t) union synthetic_ground_state(t)`

The following recursive form is prohibited:

`output(t) = output(t-1) + global_map(t) + current_ground_patch(t)`

### 8.1 Authority Rules

- Real ray-tracing points have higher authority than synthetic ground.
- If a real ground point exists near the synthetic height in the same XY cell, omit the synthetic
  point.
- If a real point indicates a low obstacle in the robot collision-height band, omit the synthetic
  point from that XY cell.
- If real points indicate a vertical wall or a terrain discontinuity, omit synthetic ground from the
  conflict region and its configured safety margin.
- A high overhead point may coexist with ground only when its clearance exceeds the shared M20
  overhead-clearance threshold.
- Never use synthetic ground to remove a real point from the global snapshot.

These rules are required because `height_cost_occupancy` uses the minimum and maximum height in each
XY cell. Blindly inserting a floor beneath a wall or person can make the vertical height gap exceed
`can_pass_under` and incorrectly classify the obstacle as passable.

### 8.2 Final Deduplication

- Synthetic state is deduplicated by 2D key `(ix, iy)`.
- The merged output is deduplicated by a 3D voxel key `(ix, iy, iz)`.
- When real and synthetic points map to the same 3D key, retain the real point.
- Build a hash set of real 3D keys and append only missing synthetic keys.
- Do not use repeated `PointCloud2.__add__` operations as an accumulator.

The upstream ray-tracing output is already voxel centered, so matching the same fixed world lattice
will minimize near-duplicate points.

### 8.3 Non-Destructive Fusion Implementation

The upsampling module may only add points to the ray-tracing result. It must not overwrite, move,
average, or delete any real point from `RayTracingVoxelMap.global_map`. The implementation must use
the following order:

1. Treat the current `global_map` as read-only. Do not mutate its NumPy array, Open3D tensor, or
   `PointCloud2` object in place.
2. Initialize the output with the complete real point set. Real coordinates, point count, and 3D
   voxel keys must remain unchanged.
3. Build the real 3D voxel-key set and XY obstacle-conflict mask from the complete real point set.
4. Validate every synthetic ground point. Discard synthetic points that duplicate a real key or fall
   inside a conflict region.
5. Apply the current global-map outer-boundary clip only to upsampled synthetic ground: delete current
   and historical synthetic points that cross it. Real ray-tracing points are not clipped and must be
   preserved unchanged. Unknown cells inside the boundary are not discarded solely because they lack
   real point coverage.
6. Append only points that pass boundary and conflict validation after the real point set, then create
   a new `PointCloud2`.
7. Do not apply column carving, voxel-centroid averaging, random downsampling, or any other operation
   that can alter real points to the combined real and synthetic set.
8. Persist only synthetic ground cells in the ground store. Never store or update real ray-tracing
   points there.

Fusion must preserve this set invariant: every real 3D voxel key remains present in the output, and
every newly added output key comes from a synthetic ground point that passed conflict validation. If
synthetic processing fails, the module must still be able to publish the current real `global_map`
unchanged.

## 9. CostMapper Integration

`CostMapper` currently consumes `RayTracingVoxelMap.global_map` by matching the `global_map` stream
name. The blueprint must explicitly remap `CostMapper.global_map` to
`global_map_upsample_ground` so the direct and augmented producers are not ambiguous.

The original `RayTracingVoxelMap.global_map` remains available for:

- Rerun visualization;
- before-and-after comparison;
- map quality metrics;
- disabling the feature without changing RayTracing behavior.

The first implementation should be guarded by an M20 blueprint configuration flag so the original
pipeline remains a fallback.

## 11. Failure and Fallback Behavior

| Failure | Required behavior |
|---|---|
| Odometry source unavailable or stopped | Skip the current ground update and merge the current real global map with existing historical synthetic ground; a temporarily empty queue follows the synchronization wait rule |
| Auxiliary queue tail is still earlier than the reference | Retain the reference global map and wait; do not estimate ground |
| Auxiliary queue head leads the reference by more than `0.05 s` | Do not update synthetic state; immediately merge the current real global map with existing historical synthetic ground, publish, and consume the reference |
| Too few candidates | Skip ground update; retain previous synthetic state |
| Plane-quality rejection | Skip ground update; retain previous synthetic state |
| Empty or failed outer boundary | Emit no synthetic points; publish real global map unchanged |
| State capacity reached | Stop inserting new cells and emit a throttled error |
| Frame mismatch | Reject merge and emit a throttled error |
| Internal exception | Publish real global map unchanged when possible |

The module must fail open to the original real map. It must never block costmap publication because a
ground fit failed.

## 12. Concurrency and Backpressure

- Store current frame, global map, and odometry in three independent bounded timestamp queues.
- Use the global-map queue head as the synchronization and output reference; attempt to advance the
  synchronization state machine whenever a queue changes.
- Protect ground-state updates and snapshot reads with a lock.
- Perform plane fitting outside the lock, then apply accepted cell updates in one short critical
  section.
- Do not silently replace a global map that is waiting for synchronization with latest-wins
  backpressure. If queue capacity forces removal of the oldest reference, first merge it with existing
  historical synthetic ground and publish `global_map_upsample_ground`, without current-frame ground
  completion or state updates.
- Prevent two fits from updating the same ground state out of timestamp order.
- Do not retain full historical point clouds in memory.

## 13. Observability

The module must publish or log the following metrics:

- Current-frame point count.
- Candidate point count after ROI and height filtering.
- Dominant Z bin and adjacent-bin counts.
- Plane inlier count and ratio.
- Plane RMSE.
- Supported XY cell count.
- Global-map boundary input count, area, and connected-region count.
- Current and historical synthetic cells suppressed for lying outside the explored global-map boundary.
- Inserted, fused, replaced, rejected, expired, and conflict-suppressed ground cells.
- Total persistent ground-cell count.
- Real, synthetic, deduplicated, and final output point counts.
- Current/global/odometry timestamp gaps.
- Lengths of all three synchronization queues and counts of successful, waiting, and unsynchronizable
  outcomes.
- Candidate filtering, fitting, state update, fusion, serialization, and total latency.
- Rejection reason counters.

Rerun should display `ground_patch` separately from the real global map using a distinct color and
allow the augmented map to be toggled independently.

## 14. Performance Budget

Initial budgets at the expected M20 point-cloud rate are:

| Metric | Target |
|---|---:|
| Ground estimation p95 | At most `50 ms` |
| State update p95 | At most `10 ms` |
| Real/synthetic fusion p95 | At most `50 ms` |
| End-to-end module p95 | At most `100 ms` |
| Synthetic cells per fully supported 5 m square | Approximately `10,000` maximum |
| Persistent memory | Bounded by `max_ground_cells` |
| Output growth after repeated traversal | Converges rather than growing linearly |

If rebuilding the full augmented cloud dominates runtime, optimize key generation and set membership
before increasing thread count. Avoid sorting-based `numpy.unique` on the full map when an O(N) hash
union is sufficient.

## 15. Test Plan

### 15.1 Unit Tests

- World-grid key generation for positive and negative coordinates.
- Repeated insertion at one XY produces one stored cell.
- Consistent revisits fuse height without increasing state size.
- Inconsistent revisits reject or replace according to confidence.
- Observation count and confidence saturation.
- Real 3D voxel wins over a synthetic duplicate.
- Global-map points outside the configured height band do not contribute to the explored boundary.
- Neither current nor historical synthetic points outside the boundary enter the merged output.
- Concave boundaries and large internal holes are not filled by an unconstrained convex hull.
- An empty or invalid boundary produces an output containing only the real global map.
- The fusion function does not mutate the input `global_map` object or its underlying point array.
- Real point count, coordinates, and 3D voxel keys are identical before and after fusion.
- A real/synthetic conflict removes only the synthetic point and never removes or moves a real point.
- Low-obstacle conflict suppresses synthetic ground.
- An auxiliary queue that brackets the reference selects the closer of the preceding and following
  samples.
- An auxiliary queue whose tail is still earlier than the reference waits without consuming the
  reference global map.
- An auxiliary queue head up to `0.05 s` later than the reference is accepted. A larger lead does not
  update state, but output still contains the current real global map and existing historical synthetic
  ground.
- Out-of-order arrivals in all three queues are still matched by message timestamp.
- Capacity limit prevents unbounded insertion.
- Empty and invalid point clouds fall back to the real map.

### 15.2 Ground Estimator Tests

- Flat ground with Gaussian noise.
- Flat ground with isolated points below the surface.
- Sparse scan lines that do not provide sufficient XY coverage.
- A low obstacle on otherwise flat ground.
- A person crossing the fit region.
- A wall intersecting the ground region.
- A hole or drop-off that must remain unsupported.
- No visible ground.
- Two separated planar patches where only the robot-connected patch is accepted.

### 15.3 Integration Tests

- Build an `UpsampleGround` module with in-memory transports.
- Feed synchronized current-frame, global-map, and odometry messages.
- Verify output timestamp and frame propagation.
- Verify the merged map contains real points unchanged.
- Verify synthetic ground fills only supported gaps.
- Verify every synthetic point is inside or on the global-map outer boundary; unknown cells inside it
  may contain synthetic points.
- Verify `CostMapper` receives the augmented stream through explicit remapping.
- Verify disabling the feature restores the original direct global-map path.

### 15.4 Replay Validation

- Run a fixed M20 recording through baseline and augmented pipelines.
- Compare known, free, occupied, and unknown costmap cell counts.
- Compare raw global-map and augmented-map point growth.
- Replay the same route multiple times and verify ground-state convergence.
- Inspect walls, stairs, ramps, people, reflective surfaces, and blind spots in Rerun.
- Record all fitting and rejection metrics for parameter selection.

### 15.5 Hardware Validation

- Stationary robot on flat ground.
- Repeated loop through the same area.
- Slow and fast traversal.
- Person crossing nearby.
- Low obstacle and wall approach.
- Ramp and threshold crossing.
- Deliberate view of a drop-off or missing-ground region under controlled conditions.

## 16. Acceptance Criteria

- All three inputs are matched in ascending message-timestamp order using the global-map queue head as
  the reference.
- When an auxiliary queue brackets the reference, synchronization selects the closer preceding or
  following sample.
- When an auxiliary queue has no sample later than the reference, the reference waits and ground
  estimation does not run.
- When an auxiliary queue head leads the reference by more than `0.05 s`, the current frame is not
  completed and synthetic-ground state remains unchanged, but `global_map_upsample_ground` still
  contains existing historical synthetic ground.
- A world XY cell contains no more than one synthetic ground point.
- Repeating the same route does not cause linear synthetic-point growth.
- After convergence, another identical traversal changes total synthetic-cell count by no more than
  5%, excluding genuinely new observed support.
- Real global-map points are byte-for-byte or numerically preserved within float32 tolerance.
- The output contains every real 3D voxel key from the input global map, with no missing keys.
- Upsampling never overwrites, averages, moves, or removes a real RayTracing point.
- Every output synthetic point lies inside the global-map outer no-go boundary projected from eligible points
  in the `[relative_z_min_m, +0.50 m]` height band.
- Persistent historical synthetic points outside the current boundary do not enter the output.
- No synthetic point is fed into `RayTracingVoxelMap`.
- Plane fits outside configured residual, inlier, or coverage limits add no ground.
- Manually identified obstacles and drop-offs are not converted into traversable filled regions.
- The augmented costmap increases known ground coverage without reducing obstacle recall in the replay
  validation set.
- Module p95 processing time meets the performance budget.
- Disabling the module restores the existing M20 navigation behavior without other code changes.

## 17. Implementation Layout

The proposed package layout is:

| File | Responsibility |
|---|---|
| `upsample_ground/__init__.py` | Public package exports |
| `upsample_ground/module.py` | DimOS module, streams, synchronization, lifecycle, configuration |
| `upsample_ground/estimator.py` | Candidate filtering, histogram seed, plane fitting, support mask |
| `upsample_ground/ground_store.py` | Persistent XY-keyed ground state and confidence updates |
| `upsample_ground/fusion.py` | Non-destructive real/synthetic conflict handling and 3D voxel union |
| `upsample_ground/test_estimator.py` | Ground estimator unit tests |
| `upsample_ground/test_ground_store.py` | Deduplication and revisit tests |
| `upsample_ground/test_fusion.py` | Authority and obstacle-conflict tests |
| `upsample_ground/test_module.py` | Stream synchronization and integration tests |

Use NumPy for array filtering and key generation. A Python dictionary is sufficient for the first
single-level implementation and makes confidence metadata explicit. The column-carving behavior in
[`voxels.py`](/dimos/mapping/voxels.py) is a useful reference, but the combined real and synthetic
map must not use column carving because it could erase real obstacles.

## 18. Delivery Phases

### Phase 1: Offline Estimator

- Implement deterministic filtering, plane fitting, support masking, and synthetic point generation.
- Add synthetic unit-test scenes and replay inspection tooling.
- Calibrate relative Z bounds and fit-quality thresholds from recorded M20 data.

### Phase 2: Persistent Ground Store and Fusion

- Implement XY-keyed state, confidence fusion, replacement, and limits.
- Implement real-point authority, obstacle conflicts, and 3D deduplication.
- Prove repeated traversal convergence with tests.

### Phase 3: DimOS Module

- Add typed streams, timestamp synchronization, backpressure, metrics, and fallback behavior.
- Add the optional diagnostic `ground_patch` output.
- Verify lifecycle and in-memory transport tests.

### Phase 4: Blueprint Integration

- Add the module to `m20_simple_nav` behind a disabled-by-default flag.
- Remap `CostMapper.global_map` to `global_map_upsample_ground` when enabled.
- Add Rerun entities for raw, synthetic, and augmented point clouds.

### Phase 5: Replay and Hardware Rollout

- Tune against recorded M20 scenes.
- Complete controlled hardware scenarios.
- Enable by default only after obstacle and drop-off acceptance criteria pass.

## 19. Key Design Decisions

- Single-level ground permits a unique `(ix, iy)` state key.
- The real global map is authoritative and stateless inside `UpsampleGround`.
- Only synthetic ground persists across frames.
- Merged output is rebuilt from current inputs; it is never recursively accumulated.
- Plane fitting requires robust geometric validation.
- Ground generation is bounded by observed spatial support.
- Synthetic ground is consumed by `CostMapper`, not by `RayTracingVoxelMap`.
- Synchronization uncertainty creates no new current-frame ground, while confirmed historical
  synthetic ground remains in `global_map_upsample_ground`.
