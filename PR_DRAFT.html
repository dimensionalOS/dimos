cc/learning · imitation learning stack on Dimos

Three drivers, one coordinator: teleop, recorder, policy.

The same Piper coordinator port shape is driven by a human (Pink IK teleop), observed by a recorder (per-episode .rrd), and re-driven by a learned policy (PolicyModule + LeRobot backend). One RobotContract converts both the recorded frames into training data and the policy's action vector back into a JointState command — that symmetry is what closes the imitation-learning loop without bespoke glue.

Hot path. Quest pose enters ArmTeleopModule, lands on the coordinator's cartesian_command port as PoseStamped, is arbitrated to the highest-priority claiming task, solved by a Pink IK QP, capped by a per-tick joint-delta safety, and published every 10 ms by the tick loop as JointState. The Piper adapter writes to CAN with the correct gripper stroke units.

  • One QP per tickDAQP differential IK on Pinocchio's forward-kinematics model — solver swap is configurable.
  • Per-tick safetymax_joint_delta_deg rejects wild commands; timeout rejects stale targets.
  • Same arbitration as beforePink IK plugs into the existing priority-based ControlCoordinator — no new arbitration logic.
  • Catalog presetsOne SingleArmPinkIKTask runtime + per-robot config presets in robot/catalog/.

Two sinks, two consumers of one config dict. Every recorder input is a typed In[T] stream slot — image, joint_state, desired_joint_action — mirroring how PolicyModule is wired. The bridge still consumes visual_override / topic_to_entity for live viewing; the recorder picks the on-disk subset (camera_key, path/metadata factories) from the same dict.

  • Pure typed inputsNo pubsubs, no topic_to_entity, no visual_override on the recorder — every input arrives on an In[T] slot.
  • Policy-aligned schemaThe recorder's image + joint_state slots match PolicyModule's, so the same producer wiring feeds both.
  • Operator boundaryQuest B button toggles episodes; empty episodes are auto-discarded; debounce eats bounces.
  • Failure isolationViewer crash does not stop recording; recorder crash does not stop viewing.

Runtime. PolicyModule runs inference on its own thread, assembles a PolicyObservation from the same streams the recorder sees, calls the configured backend, and publishes the action as JointState on joint_command — the same shape teleop produces. The low-priority policy_servo_arm consumes it; teleop always wins arbitration when an engage button is held.

Offline training prep. The same PiperRobotContract that the runtime uses to convert action vectors back into JointState is also what tells the offline converter how to read recorded entity paths. One contract, both directions — no train/deploy schema drift.

  • Same surface as teleopDownstream code can't tell whether a human or a model is in control.
  • Mid-inference preemptEngagement re-checked after select_action() returns — buttons during long inference still preempt.
  • Rollout gateOne Quest button toggles publication; both transitions reset the backend so no buffered chunk survives.
  • Priority-safe blueprintpolicy_servo_task_config() auto-derives a priority strictly below every overlapping teleop task.
  • Closed-loop test policyCaptures the live joint pose on each reset() so handoff is jump-free without a model.
  • Built-in shakedownteleop-quest-piper-policy-test exercises the full pipeline without loading lerobot/torch.

Why it composes. The teleop pillar produces cartesian_command at controller rate; the policy pillar produces joint_command at inference rate. Both flow into the same ControlCoordinator, which arbitrates by priority — the Pink IK teleop task wins whenever the operator's engage button is held. The recording pillar is a passive observer of joint_state + color_image; the same PiperRobotContract that converts recorded frames for training is what converts inference outputs back into a coordinator-native JointState. That symmetry is what closes the loop.
