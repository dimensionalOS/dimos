How every e2e test works

There is one shared harness (conftest.py). A pytest fixture launches the actual CLI — dimos --simulation <backend> run <blueprint> — as a subprocess, exactly as a user would. The test then observes and drives the running robot entirely through its real interfaces:

lcm_spy subscribes to LCM topics and records traffic; assertions are things like wait_until_odom_position(x, y) or wait_for_message_result(topic, …). human_input publishes plain-English commands to /human_input, which the in-graph LLM agent turns into tool calls. Every test first waits on /rpc/McpClient/on_system_modules/res — the signal that the agent has discovered its skills and the robot is ready.
