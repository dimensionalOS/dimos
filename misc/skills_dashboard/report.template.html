Wrap-up report

Four skills shipped. One foundation. Three honestly parked.

An eight-item wishlist for giving a robot a sense of the space it moves through — mapping it, cataloguing what's in it, reasoning about it over time. This is where the work landed: four agent-callable skills implemented and tested, a 3D scene model underpinning them, and a three-item activity track put on hold for a concrete, fixable reason rather than left ambiguous.

What shipped

The through-line

One recorded (or live) lidar+camera session flows through all four skills. Item 8's SceneModel turns the raw cloud + trajectory into a queryable 3D model; item 1 slices it into an architectural floorplan; item 2 detects, world-localizes and catalogs every object the robot passed, fusing in named places from spatial memory; item 5 asks a VLM to relabel the ambiguous ones; item 3 exposes the cloud as numpy signals (clearance, coverage, extents) for any script or skill.

How it was validated

  • 10/10 headless skill-verification tests pass — each drives the real skill code, not a mock.
  • Item 2 is replay-validated end-to-end over a recorded 87s Go2 session (accumulate → dedup → world-localize → catalog → VLM-refine).
  • Item 1 runs on real china-office lidar — the DXF / JPEG sheet in its tab is generated, not mocked.
  • Every skill is reachable as an MCP tool through the agentic Go2 blueprints.

What was parked, and why

Cut decision — activity track (items 6 → 7, and 4 alongside)

Activity recognition needs either a rigged, animated human asset in the simulator (the repo has exactly one usable human model, unrigged — every activity renders as the same static pose) or a real annotated activity dataset. Neither is on hand. Rather than ship a recognizer that can't work, the track is on hold. What was built — a deterministic DimSim scenario that spawns furniture-anchored NPCs on a seeded task schedule and drives a robot patrol to observe and label them — passes end-to-end and produces the labeled recordings a future recognizer would train from. It's plumbing proven, not effort lost.
Item 01

Architectural floorplan generation

Implemented

Give it a recording of a walk-through; get back a multi-story CAD drawing set of the building.

What it does

Multi-story architectural drawing sets — DXF with AIA layers + JPEG sheets — from lidar, live or from a .rrd recording. Wrapped as an agent-callable skill that generates the drawings and returns measured attributes of the navigated space: levels + floor-finish-levels, wall / door / stair counts, glazed fraction, mezzanines and sunken areas, extents. Debug artifacts and 3D output are off by default for MCP callers and cleaned up automatically; stylized AI renditions (drafted / cyanotype / presentation) are attached to the result.

generate_floorplan(rrd_path?, duration?, explore?, …)

dimos/mapping/floorplan/generator.py · skills/mapping/floorplan_skill.py

Evidence & next step

✓ passing1 / 1real-data e2e (china office)

Built on item 8. Every per-level evidence raster is a horizontal_section() cut of the shared SceneModel, then floor segmentation, wall-vs-furniture by height, glazing / doors / stairs, and a defect-gated cleaning pass draw the sheets.

Next: compose the skill into the agentic Go2 blueprints so it's live over MCP on-robot.

Item 02

Identify, list & locate observed items

Implemented

Navigate a space, then ask "what have you seen, and where is it?" — no target object required.

What it does

Open-vocabulary scene inventory. The robot detects whatever it sees, localizes each detection in the world frame (depth or a lidar projector for cameras without depth, like the Go2), deduplicates repeat sightings into a persistent object database, and answers as a prompt-free skill. New this pass: the catalog fuses in spatial memory — folding the robot's named/tagged places ("kitchen", "charging dock") into the inventory and tagging each object with its nearest place.

catalog_scene()locate(query) list_observed_items()detect(prompts)select(track_id)

dimos/perception/object_scene_registration.py

Evidence & next step

✓ passing2 / 2+ full replay validation

Replay-validated over a recorded 87s Go2 session. The replay surfaced and fixed two production bugs: ObjectDB TTLs running on wall-clock instead of stream-time, and a crash on coplanar (floor-plane) lidar clusters.

Next: live validation on the Go2, then feed item 4 (temporal counts) from the ObjectDB.

Item 03

Convert lidar to numpy signals

Implemented

Turn the live lidar stream into plain numpy — and into the handful of measurements skills actually ask for.

What it does

A client-side LidarPointCloudClient lets any script outside the module graph subscribe to live lidar over LCM and pull the accumulated cloud as a plain (N,3) float32 array — no hand-rolled subscribe / accumulate / vstack bookkeeping. New: a pure-numpy signals layer turns that array into the products the objectives need — measure the space, reactive clearance / safety, a portable occupancy grid, and a coverage report — with the four highest-value queries exposed as MCP skills.

measure_space()nearest_obstacle_distance() check_clearance(heading_deg, fov_deg)coverage_report()

dimos/mapping/pointclouds/live.py · signals.py · skills/mapping/lidar_signal_skills.py

Evidence & next step

✓ passing35 / 35unit tests

Promoted this sprint from a library-only utility to an implemented skill. Item 1's floorplan collector already consumes the client.

Next: validate the four MCP skills via the master-VQA e2e (its question bank now includes measure / clearance / coverage questions).

Item 05

Contextual labeling — educated guesses

Implemented

When the detector shrugs ("object", low confidence), ask a VLM — and give it the context the detector never used.

What it does

When the detector is unsure or produces a useless label, ask a vision-language model for an educated guess — feeding it evidence the detector ignores: the cropped detection image, the detector's own guess + confidence, the object's measured real-world size, and what else is nearby. A "mouse" next to a keyboard and a monitor is a computer mouse, not an animal.

identify_object(object_id)refine_observed_labels()

dimos/perception/contextual_labeling.py

Evidence & next step

✓ passing2 / 2+ real moondream VLM run

Exercised for real against item 2's replayed session — all 16 catalogued objects refined, descriptions landing in the list_observed_items() output.

Next: external data sources (OSM lookups, object databases) as additional context, and live validation on the Go2.

Item 08

Full 3D scene model with cross-sections

Prototype

The shared 3D substrate the floorplan is drawn from — sliceable into plan cuts and elevations.

What it does

SceneModel fuses everything a mapping session produced — the lidar cloud and robot trajectory from a Rerun recording, plus optionally the recorded camera stream for color — into one queryable 3D model, sliceable into horizontal plan cuts and vertical elevations. Exports to interactive viewer, mesh (Poisson → GLB/OBJ/PLY/STL with vertex colors), and voxel grid. It's the foundation item 1 builds on.

dimos/mapping/reconstruction/scene_model.py · demo: scripts/demo_rrd_3d_model.py

Why prototype, not shipped

The library API is complete and unit-tested, but there's no agent-callable skill wrapper yet — so it's honestly a prototype, not a shipped skill. It's already load-bearing: item 1's floorplan pipeline slices this model first, then draws.

✓1 / 1sections + exports test

Next: wrap "model the space seen so far" / "show me a section at z = 4.8" as a skill.

Items 04 · 06 · 07

The activity track — on hold

On hold

Parked deliberately, for a specific and fixable reason — not abandoned, not half-shipped.

The reason

Recognizing activities (not just static objects) needs either a rigged, animated human asset in DimSim — the repo has one usable human model, unrigged, so every activity renders as the same static pose — or a real annotated activity dataset. Until one exists, a recognizer can't be trained or honestly validated, so items 6 and 7 are held. Item 4 (temporal item-counts) is held alongside them: it's far more useful once activities, not just items, can be counted the same way.

Item 4 — Temporally count items

How many chairs today vs. yesterday; how often the cup moves. No recognizer yet, but item 2 laid the groundwork — ObjectDB already dedups over time and Object.detections_count is a running per-item count.

Item 6 — Identify specific activity

Action recognition over a temporal window — what is happening, not what is there. Needs a temporal model and training data. Held on the asset/dataset gap above.

Item 7 — Count activity over time

Aggregate activity events into rates — the activity analogue of item 4. Purely downstream of item 6; nothing to build until 6 resumes.

What was built anyway (and passes)

A deterministic DimSim scenario: furniture-anchored NPCs on a seeded task schedule, a robot activity-patrol that drives to observe each, recording what it sees (.rrd) plus ground-truth labels. Proves the scene / schedule / patrol machinery end-to-end and generates the labeled data a future recognizer trains from.
