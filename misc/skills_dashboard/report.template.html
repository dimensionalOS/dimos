How the pieces connect

One recorded (or live) lidar + camera session drives all four skills off one representation. SceneModel (item 8) ingests the Rerun recording's Points3D batches and trajectory strips into a queryable 3D model. Item 1 slices that model into per-level occupancy rasters (horizontal_section), classifies wall-vs-furniture by height, vectorizes wall runs / doors / glazing, and writes a layered DXF + JPEG set. Item 2 runs YOLO-E per frame, back-projects each 2D detection through the world-frame lidar cloud (Object.from_2d_to_list_lidar), and dedups repeat sightings into a persistent ObjectDB. Item 5 sends only the ambiguous detections to a VLM with measured size + neighbor context. Item 3 exposes the same cloud as an (N,3) numpy array plus stateless signal functions (extents, clearance, coverage).

Verification

  • 10/10 skill-verification tests pass; each drives real skill code, not a mock.
  • Item 2 replay-validated over a recorded 87 s Go2 session: 16 permanent objects, each corroborated ≥6×; ≤20% of ≥100 frames dropped by camera/lidar alignment.
  • Item 3 ships 35 passing unit tests over the client + pure-numpy signals layer.
  • Item 1 runs on real china-office lidar; the DXF/JPEG sheet in its tab is generated, not mocked.

Two production bugs the replay surfaced & fixed

  • ObjectDB TTLs ran on wall-clock, not stream-time — replayed detections timestamped months in the past were pruned before they could dedup/promote. Now keyed on stream time.
  • Coplanar (all-floor) lidar clusters crashed Qhull during OBB fitting — now falls back to an axis-aligned box when the cluster is near-degenerate.

Activity recognition needs either a rigged, animated human asset in DimSim (the repo has one usable human model, unrigged — every activity renders as the same static pose) or a real annotated activity dataset. Neither is on hand, so a recognizer can't be trained or honestly validated, and items 6/7 are held; item 4 (temporal item-counts) is held alongside since it is far more useful once activities, not just items, can be counted the same way. Item 01

Architectural floorplan generation

Implemented, needs thorough testing

Recording (or live drive) of a walk-through in; multi-story DXF + JPEG drawing set out, plus measured attributes of the space.

Produces multi-story architectural drawing sets — DXF with AIA layers + JPEG sheets — from lidar, live or from a .rrd recording. The @skill returns measured attributes of the navigated space (levels + floor-finish-levels, wall/door/stair counts, glazed fraction, mezzanines/sunken areas, extents). Debug artifacts and 3D-model output are off by default for MCP callers and cleaned up automatically; stylized renditions are attached to the response as images the calling agent can actually see.

Agent skill

generate_floorplan(rrd_path?, duration?, explore?, save_3d_model?, …) Generates the DXF + JPEG set (from a recording, or exploring live first) and answers with the space's measured attributes. save_3d_model also writes the sliceable .model.rrd. dimos/skills/mapping/floorplan_skill.py

How it works

  1. Collect

    Subscribes to /lidar + /global_map via item 3's LidarPointCloudClient and tracks pose on /odom, optionally driving the robot (--explore/--spin); or loads an offline Rerun recording.

  2. Detect floor levels

    The trajectory z-histogram reveals which floors were walked. Stories must be ≥ 1.8 m apart; each level's occupancy band is capped 2.6 m above its floor so roof points drop out. One sheet per level.

  3. Rasterize & classify

    Points bin into per-level occupancy grids. A cell is structural (wall) when lidar sees returns well above furniture height; indoor vs outdoor is decided by ceiling evidence (returns above head height). Trees, vehicles, ground clutter drop out.

  4. Vectorize

    Finds the dominant wall angle, unrotates, extracts straight runs. Gaps < 0.35 m bridged as noise; runs < 0.8 m dropped; gaps of 0.55–1.25 m (or where the trajectory crossed a run) become door openings with swing arcs. Runs the lidar saw through are drawn as glazing (layer A-GLAZ).

  5. Seal room loops (voxel model)

    Item 8's VoxelOccupancy.closed_region() seals each connected area the robot drove through: progressive morphological closing picks the smallest kernel (0.3 → 2.5 m) at which the trajectory-seeded flood stops leaking, bounded by the ceiling-evidence indoor mask. Closed loops land on A-AREA; doors stay open (the trajectory proved them passable).

  6. AI review (optional) & export

    A GPT-4o pass (on when OPENAI_API_KEY is set) inspects a rendered debug image per level and can flip misclassified clusters. Writes <out>.dxf (ANSI31 wall hatch, mm dimension strings, layers A-WALL/A-DOOR/A-FURN/A-ANNO-*) and a cyanotype JPEG sheet with title block.

Item 02

Identify, list & locate observed items

Implemented, needs thorough testing

Navigate a space, then ask "what have you seen, and where is it?" — open-vocabulary, no target object required.

Open-vocabulary scene inventory. The robot detects whatever it sees (prompt-free), localizes each detection in the world frame — via depth back-projection, or a lidar projector for cameras without depth like the Go2 — deduplicates repeat sightings into a persistent ObjectDB, and answers as an agent skill. This pass adds spatial-memory fusion: catalog_scene() folds the robot's named/tagged places (which detection alone can't produce) into the inventory and tags each object with its nearest place; the link is an optional dependency, so the catalog degrades cleanly to objects-only when no spatial-memory module is running.

Agent skills

catalog_scene()One-call bundle: refine ambiguous labels via the item 5 VLM pass, fold in spatial memory's named places, tag each object with its nearest place, return the fused inventory.
locate(query)Answer "where is X?" via a three-tier fallback: matching catalogued object → semantically-matched tagged place → CLIP visual recall of the space.
list_observed_items()Prompt-free inventory: one line per item with world position, size, confidence, observation count (+ description once refined).
detect(prompts)Prompted open-vocab search, e.g. detect(["cup", "mouse"]) — returns matches with stable object_ids.
select(track_id)Promote a tracked detection to a permanent object.
dimos/perception/object_scene_registration.py · detection/type/detection3d/object.py · detection/objectDB.py

How it works

  1. Quality gate

    Each frame is checked against min_frame_brightness/min_frame_sharpness (china-office feed opts in at 0.06/0.12). Near-black or motion-smeared frames are dropped — they embed semantically close to everything — and don't become the cached "latest frame" that on-demand detect() reuses.

  2. Detect (2D, open-vocab)

    YOLO-E (yoloe-11l-seg-pf.pt, prompt-free LRPC mode) runs per frame, timestamp-aligned with the depth image or lidar cloud.

  3. Localize (depth or lidar)

    Two interchangeable paths feed the same Object type. Depth: back-project the masked RGBD region via Open3D. Lidar: project the world-frame cloud into each 2D bbox (Object.from_2d_to_list_lidar), strip occluded/stray points (raycast + outlier filters), fit an oriented box — falling back to axis-aligned when sparse near-coplanar clusters make OBB fitting fail. This is the path the figure above visualizes.

  4. Deduplicate & persist

    ObjectDB merges sightings within a dedup radius (0.2 m depth; 0.35 m for the jitterier Go2 lidar), accumulates each object's pointcloud, and promotes it from pending to permanent after 6 detections. Each merge keeps the sharpest crop (highest Laplacian variance) as the image the VLM later labels from — geometry tracks the latest sighting, appearance is quality-selected.

  5. Fuse spatial memory (optional)

    Through an optional SceneMemorySpec dependency, catalog_scene() adds a "Known places" section and a nearest-place tag per object; locate() resolves subjects in order of precision (object → place → CLIP text-to-image recall, gated by a similarity threshold). Absent a spatial-memory module the attribute stays None and every skill degrades to objects-only.

Item 03

Convert lidar to numpy signals

Implemented, needs thorough testing

Live lidar stream → a plain (N,3) numpy cloud, plus the stateless signal functions skills actually ask for.

A client-side LidarPointCloudClient lets any script or skill outside the module graph subscribe to live lidar over LCM and pull the accumulated cloud as a plain (N,3) float32 array — no hand-rolled subscribe/accumulate/vstack bookkeeping. It was extracted from the floorplan demo's ad-hoc collector, which now consumes it. On top sits signals.py: a stateless pure-numpy layer that turns that array into the products four objectives need, with the four highest-value queries exposed as MCP skills.

Agent skills

measure_space()Bounding box + dimensions (w × d × h) of the scanned space — "how big is this room?"
nearest_obstacle_distance()Horizontal distance to the closest point at body height — a quick "is anything close to me?"
check_clearance(heading_deg, fov_deg)Distance to the nearest point in a cone around a heading (default straight ahead) — "is this direction clear?"
coverage_report()Points + frames accumulated and approximate floor area scanned — "how much have you seen?"
dimos/mapping/pointclouds/live.py · signals.py · dimos/skills/mapping/lidar_signal_skills.py

How it works

  1. Subscribe

    Opens LCM subscriptions to /lidar (per-scan chunks) and optionally /global_map (already-accumulated map). Usable as a context manager.

  2. Accumulate

    Each /lidar message is converted with points_f32() and appended; empty messages are ignored. /global_map is latest-wins (cumulative upstream), not appended. All state sits behind one lock.

  3. Snapshot on demand

    snapshot() returns a thread-safe vstack of buffered chunks + global map as (N,3) float32 (or (0,3) when nothing arrived); clear() drops chunks but deliberately keeps the last global map.

  4. Derive signals (pure numpy)

    signals.py is stateless — each function takes an (N,3) array and returns numpy/scalars (no LCM, Open3D, or graph): extents(), ground_height(), nearest_obstacle(), clearance(heading, fov), occupancy_grid(), height_map(), coverage_area(). Empty-cloud policy: measurements raise, distance queries return inf (nothing near = clear), grids/area return empty/zero.

  5. Ask the live sensor

    Thin client wrappers apply the signal tools to the current snapshot, so an outside-graph caller asks the sensor directly: client.extents(), client.nearest_obstacle(height_band=…), client.clearance(heading), client.coverage_summary().

from dimos.mapping.pointclouds.live import LidarPointCloudClient

with LidarPointCloudClient() as client:   # /lidar + /global_map
    time.sleep(5.0)                        # let scans arrive
    pts = client.snapshot()                # (N, 3) float32, world frame
    heights = pts[:, 2]                    # plain numpy from here on

the floorplan collector (item 1) consumes exactly this API

Item 05

Contextual labeling — educated guesses

Implemented, needs thorough testing

When the detector shrugs ("object", low confidence, class_17), ask a VLM — with the context the detector never used.

When the detector is unsure or produces a useless label, ask a vision-language model for an educated guess — feeding it evidence the detector never uses: the cropped detection image, the detector's own guess + confidence, the object's measured real-world size (from item 2's localization), and what else is nearby. A "mouse" next to a keyboard and a monitor is a computer mouse, not an animal. Exposed as agent skills on the same ObjectSceneRegistrationModule.

Agent skills

identify_object(object_id)Refine one object's label with the VLM (crop + size + neighbors as context); the refined name persists in the ObjectDB.
refine_observed_labels()Batch sweep over every ambiguous permanent object; item 2's catalog_scene() calls this automatically.
dimos/perception/contextual_labeling.py

How it works

  1. Gate on ambiguity

    is_ambiguous() selects only detections worth a VLM call: confidence below 0.5, a generic name (object, thing, unknown), or a raw class_N id. Confident specific labels are left alone — no call spent.

  2. Build the context prompt

    build_context_prompt() assembles what the robot knows but the detector ignored: detector guess + confidence, measured world size, and up to 8 nearest neighbors from the ObjectDB as "keyboard (0.4 m away)" lines, nearest first.

  3. Ask the VLM

    contextual_label() crops the stored detection image (20 px padding), queries the configured VlModel for strict JSON, and returns a ContextualLabel (label, one-line description, clamped confidence). Query failures or malformed JSON return None — the raw detector label is kept.

  4. Persist the refinement

    Object gains refined_name / refined_description alongside the preserved raw name; encoders and find_by_name prefer the refined label, and it survives update_object() merges of new sightings.

# the detector saw a blurry blob on the desk
>>> module.identify_object("a1b2c3d4")
Refined 'object' (conf 0.31) → 'espresso machine' (conf 0.78):
  "Compact chrome espresso machine with a portafilter."

or sweep everything ambiguous at once with refine_observed_labels()

Item 08

Full 3D scene model with cross-sections

Prototype

The shared 3D substrate the floorplan is drawn from — sliceable into plan cuts and elevations, exportable to mesh/voxel/viewer.

SceneModel fuses everything a mapping session produced — the lidar cloud and robot trajectory from a Rerun recording, plus optionally the recorded camera stream for color — into one queryable 3D model, sliceable into horizontal plan cuts and vertical elevations for viewing and analysis. It is the representation item 1's floorplan generation builds on. Prototype, not a shipped skill: the library API is complete and unit-tested, but there is no agent-callable wrapper yet — though it is already load-bearing, since item 1 slices this model first, then draws.

How it works

  1. Ingest

    SceneModel.from_rrd() walks the recording's chunks collecting every Points3D batch (map) and LineStrips3D strip (trajectory), then clips z outliers by percentile. The shared loader load_rrd_points() is what item 1 now uses too.

  2. Colorize from imagery

    colorize_from_session() correlates the session's stamped poses (gtsam_odom.tum) with the recorded color_image stream, projects sampled frames through the calibrated camera, and paints each point from its nearest view (z-tested). 50% of the china-office 6.4M-point cloud colorized from 84 frames in ~30 s.

  3. Slice

    horizontal_section(z) and vertical_section(p1, p2) rasterize a thick planar cut into a Section — hit-count density plus mean RGB where colorized. An optional bounds window makes multiple sections cell-aligned so they combine directly; item 1 builds all its per-level evidence rasters this way.

  4. Voxelize & seal

    voxel_occupancy(voxel=0.1) bins the cloud into a dense hit-count grid (~20 MB for a whole office). VoxelOccupancy.closed_region() then seals each connected driven-through area via progressive morphological closing (0.3 → 2.5 m, smallest that seals wins) with a leak test, yielding a watertight interior, closed room-loop polygons, and inferred bridged cells.

  5. Export 3D

    save_rerun() writes an interactive colored model; save_rerun_sliceable() logs one horizontal section per z-step on a cut_z timeline (the viewer's time slider sweeps a cut plane); to_mesh() runs Poisson reconstruction and save_mesh() exports GLB/glTF/OBJ/PLY/STL with vertex colors.

from dimos.mapping.reconstruction.scene_model import SceneModel

model = SceneModel.from_rrd("chinaOffice.rrd")
model.colorize_from_session("~/session_dir")      # camera → RGB points
plan = model.horizontal_section(z=1.6)              # plan cut
elev = model.vertical_section((5, -35), (10, 25))    # elevation
model.save_mesh("model.glb")                          # or .obj/.ply/.stl

on the china-office scan: 3 plan cuts, a full-building elevation, and a 509k-triangle colored GLB in under a minute

Items 04 · 06 · 07

The activity track — on hold

On hold

Held on a specific asset/dataset gap — not abandoned. The data-generation harness that unblocks them is built and passing.

The blocking gap

Recognizing activities (not just static objects) needs either a rigged, animated human asset in DimSim — the repo has one usable human model, unrigged, so every activity renders as the same static pose — or a real annotated activity dataset. Until one exists, a recognizer can't be trained or validated, so items 6 and 7 are held. Item 4 (temporal item-counts) is held alongside them: it is far more useful once activities, not just items, can be counted the same way.

Item 4 — Temporally count items

How many chairs today vs. yesterday; how often the cup moves. No recognizer yet, but item 2 laid the groundwork — ObjectDB already dedups over time and Object.detections_count is a running per-item count.

Item 6 — Identify specific activity

Action recognition over a temporal window — what is happening, not what is there. Needs a temporal model and training data. Held on the asset/dataset gap above.

Item 7 — Count activity over time

Aggregate activity events into rates — the activity analogue of item 4. Purely downstream of item 6; nothing to build until 6 resumes.

What was built anyway (and passes)

A deterministic DimSim scenario: furniture-anchored NPCs on a seeded task schedule, a robot activity-patrol that drives to observe each, recording what it sees (.rrd) plus ground-truth labels. Proves the scene / schedule / patrol machinery end-to-end and generates the labeled data a future recognizer trains from.
